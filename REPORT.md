# REPORT

## 1. 최초 도메인 설계 개요

사전 제출한 UML에서는 영화 예매 도메인을 `Screening`, `SeatLayout`, `Seat`, `Ticket` 중심으로 나누고, 상영 정보와 좌석 상태, 예매 정보를 각각 별도의 객체가 담당하도록 설계했다.

특히 원시값을 직접 흩어 쓰지 않으려는 의도로 `Minutes`, `PhoneNumber`, `SeatNumber`, `SeatNumbers` 같은 값 객체를 두었고, 도메인 컬렉션도 객체로 다루려는 의도로 `Seats`, `Tickets` 같은 컬렉션 래퍼를 두었다.

## 2. UML과 실제 구현의 차이

### 2.1 SeatStatus 추가

초기 UML에서는 `Seat`가 `reserve()`와 `release()`를 직접 수행하는 것으로만 표현되어 있었다. 따라서 좌석 상태 전이 규칙은 `Seat` 내부에서 함께 다루는 구조를 전제로 하고 있었다.

실제 구현에서는 상태 전이 규칙의 책임을 분리하려는 의도로 `SeatStatus`를 추가했다. 이에 따라 `Seat`는 좌석 번호와 현재 상태를 보관하고, 상태 전이는 `SeatStatus.reserve()`, `SeatStatus.release()`에 위임하도록 구현했다.

이미 예매된 좌석의 재예매, 예매되지 않은 좌석의 취소처럼 상태에 따라 달라지는 예외 규칙도 `SeatStatus`가 처리하도록 두었다.

### 2.2 값 객체를 record 기반 불변 구조로 구체화

초기 UML에서도 `Minutes`, `PhoneNumber`, `SeatNumber`, `SeatNumbers`는 값 객체로 설계되어 있었다. 다만 UML 수준에서는 이 값 객체들이 어떤 방식으로 불변성을 보장하고 어떤 시점에 유효성 검사를 수행하는지까지는 드러나지 않았다.

실제 구현에서는 값 자체가 규칙을 가지도록 하려는 의도로 `Movie`, `Minutes`, `PhoneNumber`, `SeatNumber`, `SeatNumbers`, `Seats`를 `record` 또는 불변 컬렉션 복사 방식으로 구현했다. 그리고 생성 시점에 제목 공백 여부, 상영 시간 양수 여부, 전화번호 형식, 좌석 범위, 좌석 중복 여부를 검증하도록 두었다.

잘못된 값이 도메인 안으로 들어오기 전에 차단하려는 의도로 검증 시점을 생성자로 당겼고, 값 비교와 전달을 값 중심으로 표현하려는 의도로 `record`를 사용했다.

### 2.3 Screening과 SeatLayout의 책임 경계 구체화

초기 UML에서는 `Screening`이 `movie`, `totalSeatCount`, `remainingSeatCount`, `availableSeats`, `reserve`, `release`를 제공하고, `SeatLayout` 역시 좌석 수와 예약 기능을 제공하도록 설계되어 있었다. 즉, 상영과 좌석 배치가 협력하지만 실제 책임 분배는 구현 단계에서 조정될 여지가 있었다.

실제 구현에서는 상영 정보와 좌석 관리 책임을 나누려는 의도로 `Screening`과 `SeatLayout`의 경계를 더 분명하게 잡았다. 이에 따라 `Screening`은 상영 식별자와 영화, 좌석 배치를 소유하고, 좌석 수 계산과 좌석 예약/해제는 `SeatLayout`에 위임하도록 구현했다.

상영은 예매 대상이라는 문맥을 표현하고, 좌석 배치는 특정 상영의 좌석 집합을 다루는 역할을 맡도록 구조를 나누었다.

### 2.4 TicketStatus의 상태 전이 책임 명확화

초기 UML에는 `TicketStatus` 열거형이 존재했지만, 취소 가능 여부와 중복 취소 방지 규칙이 어디에 위치할지는 구현에서 확정해야 하는 부분이었다.

실제 구현에서는 예매 취소를 하나의 도메인 행위로 유지하면서 상태 전이 규칙의 책임을 분리하려는 의도로 `TicketStatus`를 활용했다. 이에 따라 `Ticket`은 취소 요청을 받고 본인 확인과 좌석 해제를 조율하며, 상태 변경은 `TicketStatus.cancel()`에 위임하도록 구현했다.

중복 취소 여부처럼 상태에 따라 달라지는 규칙은 `TicketStatus`가 처리하도록 두고, `Ticket`은 예매 자체의 문맥을 유지하도록 역할을 나누었다.

## 3. 변경의 방향

구현 과정에서는 객체가 자신의 규칙을 스스로 가지도록 만들려는 방향을 기준으로 구조를 구체화했다. UML에서는 도메인 개체와 관계를 먼저 잡았고, 실제 구현에서는 여기에 상태 전이 규칙, 생성 시점 검증, 책임 위임 구조를 추가했다.

도메인 규칙을 외부 절차나 분산된 조건문으로 두지 않으려는 의도로, 값 검증은 값 객체에 두고 상태 전이는 status enum에 두고 상위 객체는 협력을 조율하도록 정리했다.

## 4. SOLID 관점에서의 설계 설명

### 4.1 SRP

객체지향과 클린 코드를 위해 각 객체가 하나의 이유로만 변경되도록 역할을 나누려고 했다. `Screening`은 상영 맥락을 나타내고, `SeatLayout`은 좌석 집합 관리, `Ticket`은 예매 취소를 포함한 예매 정보 관리, `Tickets`는 예매 컬렉션 조회를 담당한다. 값 객체들은 각자의 유효성 검사를 책임진다.

특히 상태 전이 규칙의 책임을 분리하기 위해 `SeatStatus`와 `TicketStatus`를 별도의 enum으로 두었다. 이를 통해 본체 객체가 데이터 보관과 도메인 행위 조율에 집중하고, 상태 판정과 예외 정책은 status enum이 담당하도록 역할을 나누었다.

### 4.2 OCP

클린 코드 관점에서 변경 지점을 줄이기 위해 상태 전이 규칙을 enum 내부 메서드로 캡슐화했다. 좌석 상태나 예매 상태의 규칙이 달라질 때 `Seat`, `Ticket`의 핵심 흐름을 크게 건드리지 않고 상태 타입 중심으로 수정할 수 있도록 구성했다.

### 4.3 LSP와 ISP

이번 구조에서는 상속 계층이나 인터페이스 중심 다형성을 앞세우기보다, 현재 도메인 크기에 맞는 협력 구조를 유지하려는 의도로 합성과 값 객체 중심 모델링을 선택했다. 그래서 상속 구조를 늘리기보다 객체 간 메시지와 값 타입을 먼저 정리했다.

### 4.4 DIP

핵심 도메인에서는 불필요한 추상화를 늘리지 않으려는 의도로 도메인 객체들이 직접 협력하는 구조를 유지했다. 대신 입출력과 랜덤, 시퀀스 생성을 위한 `utils` 패키지를 별도 유틸 객체로 분리해 두어, 외부 요소와 도메인 규칙의 경계를 나누는 방향으로 구성했다.

## 5. 데메테르 법칙을 고려한 설계

객체가 연쇄적으로 내부 구조를 파고들기보다, 자신이 직접 아는 객체에게 메시지를 보내도록 설계하려 했다. 예를 들어 `Screening`은 좌석 관리 세부 내용을 직접 다루지 않고 `SeatLayout`에 `reserve`, `release`, `availableSeats`를 위임한다. `Ticket` 역시 좌석 번호를 직접 조작하지 않고 `seats.toSeatNumbers()`와 `screening.release(...)`를 통해 협력한다.

상위 객체가 하위 객체의 내부 상태를 직접 꺼내 판단하지 않으려는 의도로, 필요한 연산은 협력 객체의 메서드 호출로 표현했다.

현재 구현에서는 `SeatLayout`이 `seats.values()`를 통해 내부 컬렉션을 순회하며 필터링과 탐색을 수행한다. 컬렉션 내부 탐색 책임도 컬렉션 객체로 모으려는 방향을 더 밀어붙인다면, `Seats`가 좌석 조회, 예약 가능 좌석 조회, 좌석 번호 기반 탐색까지 담당하도록 확장할 수 있다.

즉, 현재 구조에서는 메시지 중심 협력을 먼저 적용했고, 이후에는 컬렉션 래퍼 쪽으로 탐색 책임을 더 이동시키는 방향을 생각할 수 있다.

## 6. 적용된 디자인 패턴

### 6.1 Value Object

원시 타입을 직접 전달하지 않으려는 의도로 `Minutes`, `PhoneNumber`, `SeatNumber`, `SeatNumbers`를 값 객체로 두었다. 그리고 식별자보다 값 자체의 의미를 드러내려는 의도로, 생성 시 유효한 값만 허용하도록 구현했다.

### 6.2 Collection Wrapper

도메인 관련 컬렉션 연산을 일반 `List` 사용 코드에 흩어 두지 않으려는 의도로 `Seats`, `Tickets`를 컬렉션 래퍼로 두었다. 이에 따라 `Tickets`는 전화번호 조회와 예매 번호 조회를 담당하고, `Seats`는 좌석 수 계산과 좌석 번호 변환을 담당하도록 구성했다.

### 6.3 State 패턴에 가까운 enum 기반 상태 전이

`SeatStatus`, `TicketStatus`는 전형적인 클래스로 분리된 State 패턴은 아니지만, 상태별 행위를 enum 상수별 메서드로 분리했다는 점에서 상태 패턴의 의도를 반영하고 있다. 각 상태가 자신에게 허용된 행위만 처리하고, 허용되지 않은 요청은 스스로 예외를 던진다.

작은 규모의 도메인에서 상태별 분기문을 본체 객체에 늘어놓지 않으려는 의도로, 상태 규칙을 enum 내부 메서드로 모으는 방식을 사용했다.

## 7. 결론

초기 UML에서는 영화 예매 도메인의 핵심 객체와 관계를 식별하는 데 초점을 두었다. 실제 구현에서는 이 구조를 바탕으로 상태 객체 도입, 값 객체의 불변성 강화, 책임 위임의 명확화 같은 조치를 추가했다.

객체가 자신의 규칙을 스스로 가지도록 만들려는 의도로, 상태 전이는 status enum에 배치하고 값 검증은 값 객체에 배치하고 상위 객체는 협력을 조율하도록 정리했다. 이를 통해 도메인 규칙의 위치를 객체 안쪽으로 모으는 방향으로 구현을 구체화했다.
