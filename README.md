## 영화 예매

### 기능 요구사항

#### 상영

- 영화관에서는 여러 상영 영화에 대해 예매를 진행할 수 있다.
- 각 상영은 다음 정보를 가진다.
    - 영화 제목
    - 상영 번호
    - 전체 좌석 수
    - 남은 좌석 수
    - 좌석 정보
- 각 상영 영화는 `A`부터 `F`까지의 좌석을 가진다.
- 사용자는 상영 중 하나를 선택한 뒤, 원하는 좌석을 직접 선택해 예매할 수 있다.

#### 예매

- 예매를 진행하려면 예매자의 **휴대폰 번호**를 입력해야 한다.
    - 휴대폰 번호는 예매 정보를 조회하는 기준으로 사용된다.
- 한 번의 예매에서는 다음을 만족해야 한다.
    - **1개 이상의 좌석**을 선택해야 한다.
    - 선택한 좌석은 모두 해당 상영의 좌석이어야 한다.
    - 선택한 좌석은 모두 **예매 가능한 상태**여야 한다.
- 예매가 완료되면 다음이 처리되어야 한다.
    - **예매 번호가 생성**되어야 한다.
    - 해당 상영의 **남은 좌석 수는 예매한 좌석 수만큼 감소**한다.
    - 선택한 좌석들은 예매 완료 상태로 변경된다.

#### 내 예매 조회

- 사용자는 자신의 휴대폰 번호를 입력해 본인의 예매 내역을 조회할 수 있다.
- 예매 내역 조회 시, 해당 휴대폰 번호로 예매한 **모든 예매 정보**를 확인할 수 있어야 한다.

#### 예매 취소

- 사용자는 **자신이 예매한 내역에 한해** 예매를 취소할 수 있다.
- 예매 취소를 위해서는 **예매 번호와 휴대폰 번호가 일치**해야 한다.
- 예매가 취소되면 해당 상영의 **남은 좌석 수는 다시 증가**한다.
- 예매가 취소되면 해당 예매에 포함된 좌석들은 다시 예매 가능한 상태가 된다.
- 이미 취소한 예매는 다시 취소할 수 없다.

#### 예외 처리 (즉시 종료)

아래 상황에서는 `IllegalArgumentException`을 발생시킨 후 애플리케이션은 종료되어야 한다.

- 존재하지 않는 상영을 예매하려는 경우
- 존재하지 않는 좌석을 선택한 경우
- 이미 예매된 좌석을 선택한 경우
- 존재하지 않는 예매를 조회 또는 취소하려는 경우
- 휴대폰 번호가 일치하지 않는 예매를 취소하려는 경우

#### 프로그램 동작

- 프로그램은 사용자의 요청을 입력받아 다음 기능을 처리하고 결과를 출력해야 한다.
    - 상영 목록 조회
    - 영화 예매
    - 본인 예매 조회
    - 예매 취소

---

### 입출력 요구사항

#### 입력

사용자는 다음 기능 중 하나를 선택할 수 있다.

1. 상영 목록 조회
2. 예매
3. 내 예매 조회
4. 예매 취소
5. 종료

기능별 추가 입력은 다음과 같다.

- **예매(2)**
    - 휴대폰 번호
    - 상영 번호
    - 예매할 좌석 목록
- **내 예매 조회(3)**
    - 휴대폰 번호
- **예매 취소(4)**
    - 휴대폰 번호
    - 예매 번호

좌석 목록은 쉼표로 구분하여 입력한다. 예: `A,C,F`

#### 출력

- 현재 예매 가능한 상영 목록
- 예매 성공 메시지 (예매 번호 포함)
- 내 예매 조회 결과
- 예매 취소 성공 메시지

#### 상영 목록 출력 형식

상영 목록 조회 시에는 각 상영의 정보를 출력해야 한다.

- 상영 번호
- 영화 제목
- 전체 좌석 수
- 남은 좌석 수
- 예매 가능한 좌석 목록

#### 예매 완료 출력

- 예매가 완료된 경우 **예매 번호를 함께 출력**해야 한다.

#### 내 예매 조회 출력

- 내 예매 조회 시에는 다음을 출력해야 한다.
    - 예매 번호
    - 영화 제목
    - 예매 좌석 목록
    - 예매 상태

## 사전 제출 요구사항

구현에 앞서, 지원자는 본인이 설계한 객체 구조를 표현한 UML을 사전 과제로 먼저 제출해야 한다.

UML에는 과제 해결에 필요하다고 판단한 주요 클래스와 클래스 간 관계가 드러나야 한다. 표현 방식은 자유이며, 클래스명, 주요 필드, 주요 메서드, 연관 관계를 식별할 수 있으면 된다.

사전 제출한 UML은 구현 이전의 설계 초안으로 간주한다. 따라서 실제 구현 과정에서 UML과 다른 구조로 변경되어도 무방하다. 다만, 실제 구현 결과가 사전 제출한 UML과 달라진 경우에는 그 변경 사항을 REPORT.md에 반드시 작성해야 한다.

REPORT.md에는 최소한 아래 내용을 포함해야 한다.
- 최초 UML에서 어떻게 설계했는지
- 실제 구현 과정에서 어떤 부분이 변경되었는지
- 왜 해당 변경이 필요했는지
- 변경 이후 구조가 어떤 장점을 가지게 되었는지

단순히 “구현하면서 수정했다” 수준의 설명은 충분하지 않다. 설계 변경의 배경과 판단 근거가 드러나도록 작성해야 한다.

평가 시에는 사전 UML과 최종 구현 결과의 동일 여부 자체보다, 설계 의도를 가지고 시작했는지, 구현 과정에서 어떤 판단으로 구조를 조정했는지, 그리고 그 내용을 명확하게 설명할 수 있는지를 함께 본다.

### 프로그래밍 요구사항

- 프로그램을 실행하는 시작점은 `Application`의 `main()`이다.
- JDK 17 버전에서 실행 가능해야 한다.
    - JDK 17에서 정상 동작하지 않을 경우 0점 처리한다.
- 자바 코드 컨벤션을 지키면서 프로그래밍한다.
- `indent`(인덴트, 들여쓰기) depth를 3이 넘지 않도록 구현한다.
    - 2까지만 허용한다.
    - 예: `while`문 안에 `if`문이 있으면 들여쓰기는 2이다.
    - 힌트: `indent depth`를 줄이는 좋은 방법은 함수(또는 메서드)를 분리하는 것이다.
- 3항 연산자를 쓰지 않는다.
- 함수(또는 메서드)가 한 가지 일만 하도록 최대한 작게 만들어라.
- 예외 상황은 `IllegalArgumentException`으로 처리한다.
    - 예외 발생 시 애플리케이션은 종료되어야 한다.
- 구현 시 입출력, 랜덤값, 시퀀스 생성이 필요한 경우 `UTILS.md`에 제공된 유틸 인터페이스를 사용한다.
- 객체의 역할과 책임이 드러나도록 구현한다.
- 테스트 코드를 작성한다.

### 추가 안내사항

- 상영 정보는 프로그램 시작 시 미리 준비되어 있다고 가정한다.
- 상영 정보의 개수와 내용은 자유롭게 구성할 수 있다.
- 예매 번호는 각 예매를 구분할 수 있어야 한다.
- 휴대폰 번호는 동일한 사용자를 식별할 수 있어야 한다.
- 예매 취소 여부를 올바르게 관리할 수 있어야 한다.
- 사용자가 잘못된 값을 입력하는 경우, 별도의 재입력을 받지 않고 예외를 발생시킨 후 종료한다.
